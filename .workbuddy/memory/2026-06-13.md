## BacklogPanel 滚轮下滚关闭

**修改文件**: `scripts/konado/backlog_panel.gd`

**变更**:
- 新增 `SCROLL_SESSION_TIMEOUT_MS = 150` 常量
- 新增 `_last_wheel_time_msec` 和 `_scroll_session_started_at_bottom` 成员变量
- `_input()` 中新增滚轮下滚处理：滚动会话检测机制
- 新增 `_is_at_scroll_bottom()` 辅助方法
- `open()` 中重置会话状态

**Bug 修复 (2026-06-13 第二轮)**:
- `_input()` 中 WHEEL_DOWN 检测去掉 `event.pressed` 判断 — 在部分 Godot 4 版本中滚轮事件的 `pressed` 字段为 `false`
- 同步修复 `test_dialogue_screen.gd` 中 WHEEL_UP 的同样问题

**交互逻辑**:
- 鼠标滚轮事件间隔 >150ms 视为新会话
- 新会话开始时记录是否已在滚动底部
- 从底部开始的会话 → 所有 wheel_down 关闭面板
- 从中间开始的会话 → 正常滚动，不关闭（防误触）

**重构 (2026-06-13 第三轮)**:
- _input() 方案仍不生效，根因：Godot 4 中 Control 节点的 _input() 与 GUI 事件系统处于不同通道，对动态添加的 Control 节点调度不可靠
- 新建 `scripts/konado/backlog_scroll_container.gd`，继承 ScrollContainer，重写 `_gui_input()`
- 滚动和关闭检测统一走 `_gui_input` 通道，不再各自单开
- 修改 `backlog_panel.tscn` 绑定 BacklogScrollContainer 脚本
- `backlog_panel.gd` 精简：移除所有滚轮相关代码，改为连接 `scroll_past_bottom` 信号

**修复 (2026-06-13 第四轮)**:
- BacklogScrollContainer 报错：`super._gui_input()` 无法调用——Godot 4 中 ScrollContainer 的 `_gui_input()` 是 C++ 实现，GDScript 无法通过 super 调用
- 改为帧级方案：`_process()` 每帧缓存 `_was_at_bottom`，`_input()` 中检测 WHEEL_DOWN 时直接读缓存值
- 一帧内所有滚轮事件共享同一 `_was_at_bottom` 值 → 等价于「会话」保护
- 删除 `backlog_scroll_container.gd`，恢复 `backlog_panel.tscn` 为纯 ScrollContainer
**优化 (2026-06-13 第五轮)**: 删除 `_process()` 逐帧轮询，改为连接 `scroll_container.gui_input` 信号——纯事件驱动，零性能冗余

**调试 (2026-06-13 第六轮)**: 增加诊断日志覆盖三个可能的入口点——`_input()`, `gui_input` 信号，`open()`。输出滚轮方向、scroll_vertical、max_value、会话状态等关键值，用于定位滚轮事件不触发关闭的根因。

**信号生命周期修复 (2026-06-13 第七轮)**: `gui_input` 信号连接从 `_ready()` 移至 `open()`/`close()`，面板关闭期间不再接收任何输入事件。

**滚动到底修复 (2026-06-13 第九轮)**: `open()` 中 2 帧等待 → 4 帧 + 双次设 `scroll_vertical`。根因：RichTextLabel fit_content 在 2 帧后 `max_value` 仅 117，布局继续扩展到 1008，`scroll_vertical` 停在 117 导致 `_is_at_bottom()` 始终 false。

## 交叉溶解白块闪烁修复

**修改文件**: `scripts/portrait/portrait_actor_base.gd`

**诊断**:
- 根因1（主因）：`fade_apply_state()` 创建新 SubViewport 后立即读取 `_vp.get_texture()` 设置 overlay，但 VP 使用 `UPDATE_WHEN_VISIBLE` 尚未渲染 → 空白纹理 → 白色方块
- 根因2（次要）：`_cleanup_crossfade()` 在快速重入时 `queue_free(_old_vp)`，但 `_texture_rect` 仍引用其纹理 → 纹理失效 → 白块
- 根因3（促进）：Tween 被 kill 时 `_texture_rect.modulate.a` 可能处于中间值

**修复**:
- 新建 VP 的 `render_target_update_mode` 改为 `UPDATE_ONCE`
- `_render_frame_to_viewport()` 后调用 `RenderingServer.force_draw(false)` 强制同步渲染
- `_cleanup_crossfade()` 移除 `_old_vp.queue_free()`，延迟到 `_on_crossfade_finished()` 释放
- `fade_apply_state()` 开头重构：先 kill tween → 清 overlay → 安全释放旧 `_old_vp` → `_texture_rect` 立即锁定到当前 `_vp` 纹理 → 再冻结 `_old_vp`
- 无需修改任何调用方（`force_draw` 是同步的，不引入 async 级联）

## 交叉溶解白块闪烁修复 — 第二轮（立绘消失修复）

**问题**: 上一轮使用 `RenderingServer.force_draw(false)` 无法触发 SubViewport 渲染，overlay 纹理为空白 → 交叉溶解结束后立绘消失。

**根因**: `force_draw` 只强制主 Viewport 绘制，不影响 SubViewport 的 render target 更新。

**修复 (2026-06-13)**:
- `portrait_actor_base.gd`: `force_draw(false)` → `await get_tree().process_frame`
- `portrait_acting_interface.gd`: `change_actor_state()` 对 `fade_apply_state` / `fade_set_character_texture` 调用添加 `await`
- `simple_portrait_actor.gd`: `fade_set_character_texture()` 对 `fade_apply_state` 调用添加 `await`
- `inline_command_processor.gd`: `_execute_change()` 对 `fade_apply_state` 调用添加 `await`

## 交叉溶解白块闪烁 — 第三轮（最终修复）

**问题**: 第二轮使用 `await get_tree().process_frame` 产生异步间隙，内联 `{change:...}` 连续命令在 await 挂起期间触发重入 → `_texture_rect` 被锁到空白 VP 纹理 → 白块固化。

**修复 (2026-06-13 第三轮)**:

`portrait_actor_base.gd`:
- 新增 `_pending_free_vps: Array[SubViewport]` 成员
- `fade_apply_state()`: 移除 `await` / 移除 `_texture_rect.texture` 切换 / 旧 `_old_vp` 改 append 到 `_pending_free_vps` 而不是立即 `queue_free`
- `_on_crossfade_finished()`: 追加遍历释放 `_pending_free_vps`
- `UPDATE_ONCE` → `UPDATE_WHEN_VISIBLE`

调用方回退 await:
- `portrait_acting_interface.gd`: 移除 `await`
- `simple_portrait_actor.gd`: 移除 `await`
- `inline_command_processor.gd`: 移除 `await`

核心原则: _texture_rect 永不指向未渲染 VP；旧 VP 延迟到 crossfade 完成且 _texture_rect 切换后才释放；零异步间隙。
